## How This Snapshot Was Created

1. Opened `src/apollo-3cs-0.004bf - eolus- v5.project` in CODESYS IDE.
2. Exported each library and the main project individually via **File → Export → PLCopenXML**.
3. Copied the resulting XML files into `exports/v1/xml/`.
4. Generated `MANIFEST.json` and `INDEX.json` from the exported files and prior PLC exploration findings.

## Node Discovery

The `INDEX.json` file contains:
- The root PLC node (`PrimaryPLC.System`) and its state at export time.
- All 59 operational children and 1 non-operational child discovered via `describe -children`.
- Live-verified signal names under `CANBusDrive/cTransA` (5 signals).
- Complete TCP endpoint map (9 endpoints, ports 49700–49899).

## Phase 2: XML Indexing

The `POU_INDEX.json` and `PROJECT_MAP.md` files are auto-generated by `scripts/index_xml.py` and provide:

- **495 POUs** indexed across all 9 XML files (431 functionBlock, 64 function).
- **146 DUTs** with base type classification (enums, structs, arrays, primitives, derived types).
- **44 unique library references** mapped with namespace and system-library flags.
- Per-file breakdown of POUs, DUTs, and library dependencies.

To regenerate these artifacts:

```bash
python scripts/index_xml.py exports/v1
```

See [Phase 2 documentation](../../docs/phase1-exports.md#phase-2-xml-pou-indexing--project_map) for schema details and limitations.

## Phase 3.5: Transitive Dependency & Cascade Analysis

The `TRANSITIVE_CLOSURE.json` and `CASCADE_ANALYSIS.md` files are auto-generated by `scripts/analyze_graph.py` and provide:

- **Transitive dependency closure** — For each object, all types it transitively depends on through multi-hop chains.
- **Impact cascade** — For each type, the full set of objects affected by a change (direct + indirect).
- **Centrality metrics** — In-degree, out-degree, and transitive impact/dependency counts for all 432 graph nodes.
- **Bridge/leaf/root classification** — Identifies coupling points, foundation types, and top-level consumers.
- **Notable dependency paths** — Shortest paths from Build objects to their transitive dependencies.

Key findings for v1:
- **`t_percentage`** has the highest transitive impact: changing it affects **102 objects** (81 direct + 21 indirect).
- **`SystemBuild`** is the most dependent object: it transitively depends on **51 types** through 8 direct edges.
- **`Movement`** is the top bridge node (fan-in=5, fan-out=8), making it a key architectural coupling point.
- The longest notable dependency path is **3 hops** (e.g., `ReportableChannelManagerBuild` → `ConnectionHandler`).

To regenerate these artifacts:

```bash
python scripts/analyze_graph.py exports/v1
```

See [Phase 3.5 documentation](../../docs/phase1-exports.md#phase-35-transitive-dependency--cascade-analysis) for schema details and limitations.

## Phase 4: Implementation-Level Dependency Extraction

The `IMPL_DEPS.json` and `IMPL_DEPENDENCY_MAP.md` files are auto-generated by `scripts/extract_impl_deps.py` and provide:

- **3,428 ST bodies analyzed** across all 9 XML files (POU bodies, methods, property accessors).
- **462 POUs** with implementation-level dependencies discovered.
- **1,129 FB/function calls**, **921 method calls**, **1,552 property accesses**, and **937 type casts** found.
- **173 unique types** referenced in implementation code; **122 of these are referenced ONLY in implementation** (not declared in any interface).
- Per-POU breakdown showing which types are called, accessed, or cast in each code body.
- Structural vs behavioral comparison table distinguishing Phase 3 (interface) from Phase 4 (implementation) dependencies.

Key findings for v1:
- **`SystemState`** is the most-referenced type in implementation code (38 POUs), despite not being declared as a variable type in any interface.
- **`strcmp`** (from common.xml) is used in 23 POUs' implementation — a widely-used helper function not visible from interface analysis alone.
- **122 types** are referenced in ST code but never appear in interface declarations, revealing hidden coupling that Phase 3 cannot detect.
- **`CANOpenMasterDriveBuild`** calls 22 different `map*` FBs in its implementation, showing the full device mapping scope.

To regenerate these artifacts:

```bash
python scripts/extract_impl_deps.py exports/v1
```

See [Phase 4 documentation](../../docs/phase1-exports.md#phase-4-implementation-level-dependency-extraction) for schema details and limitations.

## Phase 4.5: Unified Dependency Graph

The `UNIFIED_DEPS.json` and `UNIFIED_DEPENDENCY_MAP.md` files are auto-generated by `scripts/unify_deps.py` and provide:

- **4,209 unified edges** merging interface-derived (534) and implementation-derived (3,699) dependencies.
- **Provenance tracking** on every edge: interface-only (510), implementation-only (3,675), or both (24).
- **Combined transitive analysis** — forward closure and impact cascade across both sources.
- **Key object analysis** for parser/server/system/etc. objects with unified dependency views.
- **Combined centrality rankings** with separate fan-in/fan-out for interface and implementation.

Key findings for v1:
- **Implementation dominates** — 87% of edges come only from implementation scanning.
- **24 edges confirmed by both sources** — the most reliable dependencies in the graph.
- **1,341 types referenced only in implementation** — many are false positives (filtered in Phase 5).
- **`SystemState`** has 37 implementation fan-in but 0 interface fan-in — impact is entirely behavioral.

To regenerate these artifacts:

```bash
python scripts/unify_deps.py exports/v1
```

See [Phase 4.5 documentation](../../docs/phase1-exports.md#phase-45-unified-dependency-graph) for schema details and limitations.

## Phase 5: Graph Cleaning & Confidence Scoring

The `CLEAN_DEPS.json` and `CLEAN_DEPENDENCY_MAP.md` files are auto-generated by `scripts/clean_graph.py` and provide:

- **1,009 clean edges** after removing 3,200 false positives (76% reduction).
- **Confidence scores** on every edge: high (24), medium (510), low (475).
- **7 filter categories**: local variable names (1,070), unknown types (1,028), common member names (424), conversion functions (226), standard library (216), IEC keywords (126), self-references (110).
- **512 clean nodes** (down from 1,805 in the unified graph).
- **High-confidence edges** — 24 edges confirmed by both interface and implementation.

Key findings for v1:
- **`t_percentage`** remains the top node (fan-in=81) — the most widely-used project DUT.
- **`SystemState`** is #2 (fan-in=36) — the most impactful behavioral dependency.
- **`UInterface`** is #3 (fan-in=24) — a key architectural coupling point.
- The cleaned graph provides a **much higher signal-to-noise ratio** for impact analysis.

To regenerate these artifacts:

```bash
python scripts/clean_graph.py exports/v1
```

See [Phase 5 documentation](../../docs/phase1-exports.md#phase-5-graph-cleaning--confidence-scoring) for schema details and limitations.

## Known Gaps

1. **No method/property body parsing** — The indexer enumerates method and property names but does not parse their implementation code. *(Note: Phase 4 extracts implementation dependencies from ST bodies, but does not perform full semantic parsing.)*
2. **No variable/parameter extraction** — POU input/output/local variables are not indexed. *(Note: Phase 2.5 extracts interface variables; implementation-local variables are not captured.)*
3. **Single PLC only** — This snapshot covers only the primary PLC. No secondary PLC or remote I/O snapshots.
4. **Static state** — `INDEX.json` reflects a single point-in-time (`systemstate: charging`). Runtime state changes are not captured.
5. **No diff capability yet** — No tooling exists to compare v1 against a future v2 snapshot.
6. **Compiler defines** — Listed in MANIFEST.json but not validated against the actual compiled binary.
7. **Cross-reference scope** — Phase 3 resolves types against POU/DUT definitions within this export only. External library types (CODESYS system libraries) are not resolved.
8. **Implementation analysis is regex-based** — Phase 4 uses pattern matching, not a full ST parser. False positives/negatives are possible (see Phase 4 limitations).

## Deprecation Notice

`plc_nodes_map.json` at the project root is **deprecated** as of this export. Its content has been migrated into `exports/v1/INDEX.json`. The old file will be removed in a future phase.

## Regenerating This Snapshot

To create a new export (e.g., `v2`):

1. Open the project in CODESYS IDE.
2. Re-export all libraries and the main project as PLCopenXML.
3. Copy XML files into `exports/v2/xml/`.
4. Run `scripts/generate_manifest.py` to produce `MANIFEST.json`.
5. Run live PLC discovery to update `INDEX.json`.
6. Run `scripts/index_xml.py exports/v2` to generate `POU_INDEX.json` and `PROJECT_MAP.md`.
7. Run `scripts/build_xref.py exports/v2` to generate `XREF.json` and `DEPENDENCY_MAP.md`.
8. Run `scripts/analyze_graph.py exports/v2` to generate `TRANSITIVE_CLOSURE.json` and `CASCADE_ANALYSIS.md`.
9. Run `scripts/extract_impl_deps.py exports/v2` to generate `IMPL_DEPS.json` and `IMPL_DEPENDENCY_MAP.md`.
10. Run `scripts/unify_deps.py exports/v2` to generate `UNIFIED_DEPS.json` and `UNIFIED_DEPENDENCY_MAP.md`.
11. Run `scripts/clean_graph.py exports/v2` to generate `CLEAN_DEPS.json` and `CLEAN_DEPENDENCY_MAP.md`.
12. Update this README for the new version.
